# 产品指标看板

用途：
- 定义 A1 的产品衡量口径
- 区分“请求成功”与“产品价值成立”
- 把已有周报、评测、沙箱和脚本能力整理成可对外表达的框架

## 口径说明

这份文档不是线上经营分析，也不是用户调研满意度报告。

当前指标体系主要服务于三个目标：
- 判断 A1 的核心产品价值是否成立
- 区分“资料沉淀成功”和“复杂任务真的完成”不是一回事
- 为后续周报、沙箱验证和产品迭代提供统一口径

因此，本文中的指标分为两类：
- 已落地代理指标：当前系统已具备字段或脚本口径
- 建议补充指标：产品上很重要，但当前仍需逐步接线或补充埋点

## 一、A1 应该衡量什么

A1 当前包含两条能力路径：

1. 长期沉淀路径
   - 把视频、网页、文档、笔记等内容沉淀为可长期复用的知识资产
2. 即时处理路径
   - 对多个外部来源做当轮提取、比较和基础分析

因此，A1 的指标不能只看“有没有返回答案”，而要回答三个问题：
1. 资料有没有被成功沉淀
2. 沉淀后的资料有没有被再次调用
3. 复杂或比较任务有没有真正形成可用结果

## 二、指标结构

### 1. 沉淀层指标

回答：
`内容有没有成功进入系统，成为后续可用的知识资产。`

### 2. 复用层指标

回答：
`沉淀下来的内容，后续有没有真的被找回、被追问、被继续使用。`

### 3. 分析层指标

回答：
`系统有没有把复杂任务、比较任务和深一步处理真正交付出来。`

## 三、当前最关键的产品指标

### 3.1 北极星 1：资料二次调用率

定义：
已沉淀资料中，后续至少被一次检索、问答或复盘调用的比例。

意义：
- 这是 A1 区别于普通收藏工具的关键指标
- 如果资料从未被再次调用，沉淀价值就没有真正成立

状态：
- **已接线代理指标（最小埋点）**：用户 `commit` 入库时在 chunk metadata 打 `source_kind=user_committed`；检索命中后由 `turn_product_metrics.user_committed_retrieval_hit` 落盘 → `product_metrics.knowledge_reuse_rate` 聚合 → 周报三态判定（与北极星2 同路径）。
- 当前数据仍来自**离线沙箱代表题**；若代表题未覆盖「先 commit 再检索」链路，命中率为 0 或样本不足 → **只看趋势、不下达标结论**。

### 3.2 北极星 2：复杂任务有效完成率

定义：
在复杂问答、多来源整合、需要升级 complex 或 async 处理的任务中，最终返回“完成且可直接使用结果”的比例。

意义：
- 衡量系统有没有从“资料可问”走向“任务可分析”

状态：
- **已接线代理指标**：出口写入 `turn_product_metrics` → `scripts/report_product_metrics.py` 周报聚合，并按 `backend/config/metric_targets.py` 出三态判定（达标 / 未达标 / 样本不足）。
- 当前数据来自**离线沙箱代表题**，复杂子集样本 < 有效下限（30）→ 周报判定为「**样本不足**」，**只看趋势、不下达标结论**；线上真实流量与硬 SLO 待接量后成立。
- 运行口径：指标沙箱默认走**真实外部 LLM**（`scripts/run_metrics_sandbox.ps1` 默认 `LIGHT_MAQA_FAKE_LLM=0`）；FAKE 跑（显式 `-FakeLLM`）仅验管线连通、不计入趋势（见 `docs/evidence/known_issues.md` KI-METRICS-001）。
- **最近真实复跑（2026-06-25，`environment=REAL`，可复现于 2026-06-22 同值）三态结论**：
  - 北极星2 = **58.6%**（17/29 complex 有效完成）；
  - 判定 = 「**样本不足**」（complex 计数 **29 < 30** 下限，差 1；因 `complex_16` 本轮被判 `is_complex_task=false` 掉出复杂分母）；
  - **趋势低于 70% 目标**：即使样本补齐，58.6% 仍未达标 —— 这是**真实产品差距**，不是环境噪声；
  - 主因失败码：`upgrade_still_partial` ×6（复杂题升级 complex 后**仍停在 partial**）、`insufficiency` ×3；
  - Guardrail 同轮达标：Partial 率 20.5%（≤30 ✅）、insufficiency 占比 6.8%（≤25 ✅）、质量门通过率 77.3%。
  - 对比：FAKE 跑同口径仅 **4%** → 证实 FAKE 比率不可当产品质量（KI-METRICS-001）。
- **2026-06-26 双轮 RefineV2 沙箱（出口收口后）**：R1 **73.3%（22/30，达标）** partial 8→2；R2 **73.3%**（周报聚合仍 22/30），DIAG partial 1（`complex_03`），R2 有 `complex_04/24` **LLM 超时**（exit 1，环境方差，非逻辑回退）。`ENABLE_COMPLEX_REFINE_V2` **仍默认 OFF**（regression 33/42 未全绿）。
- **2026-06-26 R2 续跑（C1+C2+limitations，`eed5823`）**：**exit0**，TAG ok，METRICS 44/44；DIAG **complex_partial=0**；北极星2 **27/30 = 90.0%（达标）**；唯一 partial 为 `insufficiency_03`（真 insufficiency，非 complex 分母）。`complex_03/10/17` 均已 succeeded 且非 insuf。
- **S2 双轮观测（RefineV2，真 LLM）**：R1（`1d75f5f`）**23/30 = 76.7%**；R2（`eed5823`）**27/30 = 90.0%** — 连续两轮 ≥70%，partial 桶仅剩真 insufficiency。
- **RefineV2 转正后回滚触发（一行）**：若 default ON 后连续一周沙箱北极星2 **<65%**，或 complex 样本 P95 延迟 **>2×** 转正前 baseline → 将 `ENABLE_COMPLEX_REFINE_V2` default 改回 OFF，并排查 LLM/超时/样本漂移，禁止 silent 拖过两周。

## 四、分层指标建议

### 4.1 沉淀层

- 资料沉淀成功率
- 视频 / 网页 / 文档解析成功率

### 4.2 复用层

- 资料二次调用率
- 基于资料的有效回答率
- 视频复盘使用率

### 4.3 分析层

- 多来源比较完成率
- Complex 有效完成率
- Partial 率
- 复杂任务升级率
- 平均材料数
- insufficiency 占比

## 五、当前最值得看的 6 个数

如果只展示一页简洁看板，建议优先展示：
1. 复杂任务有效完成率
2. Partial 率
3. 复杂任务升级率
4. 平均材料数
5. insufficiency 占比
6. 资料二次调用率

## 六、为什么这些指标比“请求成功率”更重要

A1 的问题不是“系统能不能回复”，而是：
- 内容有没有真正沉淀下来
- 后续有没有被再次调用
- 复杂任务是不是只完成了一半
- 比较任务有没有形成真实差异，而不是摘要拼接

因此，如果只看请求成功率，会掩盖很多真正影响产品价值的问题。

## 七、v1 已落地代理指标口径

版本：`product_metrics_v1`

### 7.1 复杂任务有效完成率

分子需同时满足：
- `is_complex_task == true`
- 顶层 `task_status ∈ {done, succeeded}`
- `insufficient_evidence == false`
- `quality_gate_passed == true`

分母：
同一次响应中 `is_complex_task == true` 的 turn 数。

说明：
`partial` 不计入有效完成。

### 7.2 Guardrail 字段映射

| 指标 | 计算（v1） | 字段 |
| --- | --- | --- |
| Partial 率 | `task_status == partial` / 全部 turn | 顶层 `task_status` |
| 复杂任务升级率 | `is_complex_task` / `complex_candidate` 或 ingress 标记为 complex 的 turn | `is_complex_task` + ingress trace |
| 平均材料数 | mean(`v15_retrieved_chunks_count` + 临时材料数) | `extra.v15_retrieved_chunks_count`, `extra.v15_temporary_materials_count` |
| insufficiency 占比 | `insufficient_evidence == true` / 全部 turn | `extra.insufficient_evidence` |

### 7.3 数据来源（v1）

| 优先级 | 来源 | 说明 |
| --- | --- | --- |
| 1 | PG `turn_product_metrics` | 周报主数据源 |
| 2 | Chat 响应 JSONL 导出 | `scripts/aggregate_product_metrics.py` |
| 3 | pytest / benchmark 产物 | 离线验证，不作产品看板主数据源 |
