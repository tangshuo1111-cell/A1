# 项目卡

项目名称：
A1：面向复杂任务的多材料 AI 问答系统

项目背景：
随着 AI 应用从单轮对话走向复杂任务处理，用户的问题不再只依赖单一知识库，而是经常同时涉及网页、文档、视频和已有知识材料。传统单一 RAG 或单工具调用方式，难以稳定覆盖这类跨材料、跨步骤的问题。因此，我做了这个项目，希望验证一个能够整合多材料处理、检索、路由和生成交付的 AI 系统原型。

目标用户：
- 需要处理复杂信息任务的知识工作者
- 需要从文档、网页、视频等多来源材料中获得答案或摘要的用户
- 对答案稳定性、材料依据和任务状态反馈有要求的 AI 应用使用者

核心痛点：
- 单一知识库问答难以覆盖多来源材料场景
- 复杂问题往往需要检索、解析、比对、生成多个环节协同
- 普通问答系统即使能回答，也难以解释依据来自哪里
- 当能力增多后，系统容易出现路径不稳定、状态不清晰、结果不一致的问题

产品目标：
- 建立一个可处理知识库、网页、文档、视频等多材料的 AI 问答主链
- 让系统能够根据任务复杂度自动分流，而不是所有请求都走同一条路径
- 保证结果输出具备较稳定的状态语义、材料依据和交付质量
- 把系统从“能力堆叠”整理成“可管理的产品主链”

核心功能：
- 知识库检索与混合召回
- 网页读取与内容抽取
- 文档解析与摘要
- 视频内容提取与处理
- 复杂任务分流与多路径执行
- 多 Agent 协同处理
- 统一质量门和统一出口

业务流程：
1. 用户输入问题或上传材料
2. 系统判断请求类型、复杂度和优先路径
3. 材料进入统一预处理和检索准备阶段
4. 系统根据材料充分度和任务复杂度决定走快速回答、复杂处理或异步任务
5. 生成候选答案或任务结果
6. 最终结果经过质量门和统一出口后返回用户

AI 能力接入点：
- 语义路由
- 检索增强生成
- 多材料解析
- 材料充分度判断
- 多阶段协同执行
- 最终答案生成与质量控制

MVP 范围：
- 一个统一主入口
- 三类核心材料能力：知识库、文档、网页
- 一个稳定的默认回答路径
- 一个明确的复杂任务升级机制
- 一个统一结果出口，保证状态和返回口径一致

我的职责：
- 主导项目系统治理与结构收敛
- 梳理多模块协作边界和默认运行主路径
- 识别并收敛路由分散、状态语义不统一、材料判断重复、出口不稳定等问题
- 推动建立单一事实源、统一质量门和统一出口门
- 清理过时文档、重复治理资产和无效项目噪音
- 用 benchmark、contract test 和 smoke test 保证调整后不回退

关键难点：
- 系统能力多，链路长，容易出现重复判断和多头裁决
- 同一个问题在不同路径下可能产生不一致结果
- 多材料接入后，状态对象和材料语义容易漂移
- 项目要同时兼顾扩展性、稳定性和可解释性

解决方案：
- 将入口路由集中到统一 ingress 层
- 用共享材料预处理阶段生成统一检索快照
- 将 KB 充分度、材料充分度、最终出口分别收敛到明确模块
- 用统一 gate 管理最终路径、状态和输出语义
- 用 contract test 约束关键字段只能在统一位置写入
- 通过项目清扫和事实源压缩，降低系统认知成本和维护成本

最终结果：
- 系统主链从多能力叠加逐步收敛为统一可解释的运行路径
- 默认路径、材料流、质量门和出口语义更稳定
- 单一事实源设计已经在代码和测试层落地
- 项目仓库完成清扫和提根，具备独立展示条件
- 关键 benchmark 和核心验证链路未回退
- 产品指标 v1、用户场景复盘、评测与验收体系文档已建立（见 `docs/pm/`）

后续迭代方向：
- 将产品指标 v1 接到 JSONL/turns 自动聚合看板
- 建立更完整的线上实验闭环（非本阶段 hard SLA）
- 补齐商业化和配额策略
- 优化复杂任务下的异步体验和结果反馈
