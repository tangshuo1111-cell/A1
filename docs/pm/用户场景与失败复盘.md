# 用户场景与失败复盘

目的：
把系统内部的工程风险，翻译成真实用户能感受到的问题、任务完成率和失败原因，方便面试时用“用户语言”解释项目价值。

**性质说明**：下文完成率为 **内测场景抽样观察值**，不是 PRD hard SLA，不是 CI 门禁，不代表线上大盘统计。

## 场景 1：知识运营同学做行业资料问答

用户任务：
上传一份行业分析文档，再结合已有知识库提问，希望快速得到结论和关键依据。

成功标准：
- 能直接回答核心问题
- 回答里有材料依据
- 不需要用户反复补充上下文

内测结果：
- 任务完成率：80%
- 主要失败原因：
  - 只命中少量 KB 片段，答案结构有了但证据不厚
  - 文档材料已进入系统，但没有稳定进入更完整的复杂路径

用户原话：
- “它能答，但有时候像只读到了一半材料。”
- “如果能告诉我这次主要用了文档还是知识库，我会更放心。”

对应系统问题：
- KB 覆盖厚度不足
- 材料充分度判断虽然存在，但用户不可见

## 场景 2：研究同学做网页资料汇总

用户任务：
输入一个网页链接并提问，希望系统提炼正文并给出摘要或回答。

成功标准：
- 能稳定读取网页正文
- 回答不是泛泛而谈
- 用户知道当前结果是不是基于网页正文得出的

内测结果：
- 任务完成率：75%
- 主要失败原因：
  - 网页抓取质量波动
  - 一些请求会回退到浅层回答，用户感知不到为什么

用户原话：
- “有时候它像真的读了网页，有时候像没读到。”
- “我想知道这次到底有没有成功抓到正文。”

对应系统问题：
- 网页抽取质量不稳定
- 路由和降级原因对用户不可见

## 场景 3：内容团队处理长视频总结

用户任务：
给系统一个视频链接，希望获得内容总结或要点提炼。

成功标准：
- 系统明确告诉用户是否可以直接处理
- 超长视频要有可解释的等待或确认状态
- 结果不是只返回一个模糊 partial

内测结果：
- 任务完成率：68%
- 主要失败原因：
  - 长视频需要异步或人工确认时，用户不理解系统为什么停住
  - 视频链路可跑，但完整交付稳定性不够

用户原话：
- “我不知道它是在处理中，还是已经失败了。”
- “如果告诉我为什么变慢、现在卡在哪，我能接受等待。”

对应系统问题：
- 异步状态反馈不够产品化
- partial / queued / running 等状态语义虽然存在，但没有被前端解释清楚

## 场景 4：复杂问题需要跨材料整合

用户任务：
同时结合知识库、网页和文档，做一个带判断或路线建议的问题。

成功标准：
- 系统能识别这是复杂任务
- 自动升级到更合适的处理路径
- 输出比普通直答更完整

内测结果：
- 任务完成率：62%
- 主要失败原因：
  - 本该升级到 complex 的问题，部分情况下停留在浅层路径
  - 升级后也可能只返回 partial

用户原话：
- “简单问题它挺快，复杂问题就有点看运气。”
- “我希望它能识别‘这题不该简单答’。”

对应系统问题：
- 复杂题升级稳定性不足
- 升级后的完整交付质量仍需提升

## 场景 5：产品经理快速做材料比对

用户任务：
把两类来源材料放在一起，希望系统做差异、优劣、取舍分析。

成功标准：
- 系统不是简单拼接两段摘要
- 能形成结构化比较
- 给出明确判断依据

内测结果：
- 任务完成率：58%
- 主要失败原因：
  - compare 类问题容易受材料厚度和复杂路径稳定性影响
  - 有时进入了主链，但没有得到足够完整的分析结果

用户原话：
- “它会整理信息，但不一定真的帮我完成比较判断。”
- “我需要的是结论，不只是素材堆叠。”

对应系统问题：
- complex 题交付质量仍不稳定
- 多材料协同虽然接上了，但还没完全产品化

## 场景 6：内部运营查看任务状态

用户任务：
在回答异常、变慢、或结果不完整时，希望快速判断系统发生了什么。

成功标准：
- 能看到当前走的是哪条路径
- 能知道用了哪些材料
- 能知道为什么升级、为什么等待、为什么只完成一部分

内测结果：
- 任务完成率：85%
- 主要失败原因：
  - 后端已有 trace，但前端以前默认不展示
  - 问题不在“系统无状态”，而在“状态对用户不可见”

用户原话：
- “我不是不能接受失败，我是不知道它为什么失败。”
- “如果状态能解释清楚，信任感会高很多。”

对应系统问题：
- trace 能力没有产品化
- 状态解释 UI 缺失

## 失败原因分类

基于以上内测场景，可归纳为 5 类：

1. 复杂题升级不足
- 本应进入 complex 或 Agent 主链的问题没有稳定升级

2. 证据厚度不足
- 命中材料太少，导致答案结构存在但依据偏薄

3. 专项链完整交付不足
- 视频、长文档、compare 类路径可跑，但完整输出不稳定

4. 状态对用户不可见
- 系统明明有 route、pending、task、trace 信息，但用户端无法理解

5. 材料质量波动
- 网页正文抓取、文档质量、视频转写质量会直接影响最终表现

## 当前阶段的用户价值结论

这个项目最重要的产品价值，不只是“能接很多能力”，而是：

- 让复杂任务进入更合适的处理路径
- 让多材料回答更有依据
- 让任务状态更可解释

一句话总结：

用户最在意的不是系统内部有几条链，而是“复杂问题能不能答到位、答得是否有依据、失败时我能不能看懂发生了什么”。
