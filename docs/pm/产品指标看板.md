# 产品指标看板

目的：
为 A1 定义一页能对内沟通、对外表达的产品指标看板，区分北极星指标与 guardrail 指标。

## 一、北极星指标

### 北极星指标：复杂任务有效完成率

定义：
在复杂问答、多材料整合、需要升级 complex 或异步处理的任务中，最终返回“用户认为可直接使用的结果”的比例。

为什么选它：
- A1 的差异化不在简单问答，而在复杂信息任务处理
- 只看请求成功率会掩盖“答得浅”“只完成一半”的问题
- 这个指标能直接对应系统最核心的产品价值

计算口径建议：

复杂任务有效完成率 =
复杂任务中被标记为“完成且可直接使用”的任务数 / 全部复杂任务数

当前阶段替代口径：
- 用人工抽样打标
- 或结合 `task_status=done`、无 insufficiency 提示、质量门通过作为代理判断

## 二、Guardrail 指标

### Guardrail 1：Partial 率

定义：
所有请求中，最终只返回 `partial` 或明显不完整交付的比例。

为什么重要：
- 这是用户最直接感知到的不稳定信号
- 它通常意味着复杂题升级、材料充分度或专项链质量存在问题

建议用途：
- 用来监控交付完整性是否退化
- 不能因为追求更快而显著抬高 partial 率

## Guardrail 2：复杂任务升级率

定义：
被识别为复杂候选的问题中，最终进入 complex 或 async 路径的比例。

为什么重要：
- A1 的关键问题之一就是复杂题是否稳定升级
- 升级率过低，说明系统可能在浅层路径里“硬答”
- 升级率过高，也可能说明系统过于保守、成本过高

建议用途：
- 监控路由策略是否合理
- 和复杂任务有效完成率一起看，避免只追求升级数量

## Guardrail 3：平均材料数

定义：
每次回答实际使用的材料数量，建议至少统计：
- 检索片段数
- 临时材料数
- 已提交材料数

为什么重要：
- 材料过少时，复杂问题容易“有结构、没证据”
- 材料过多时，也可能拖慢响应或引入噪音
- 它能帮助判断系统是在“证据不足”还是“材料冗余”

建议用途：
- 观察复杂问题的证据厚度
- 与质量门通过率、任务完成率联动分析

## 三、补充观察指标

当前阶段建议同时观察：
- 请求成功率
- 平均响应时长
- 异步任务完成率
- 质量门通过率
- 网页 / 文档 / 视频三类材料的成功解析率

## 四、指标间关系

推荐看法：

- 北极星指标看“最终有没有真正解决复杂任务”
- Partial 率看“交付是否完整”
- 升级率看“路由是否合理”
- 平均材料数看“证据是否足够”

如果出现以下情况，说明系统有明显风险：

1. 升级率下降，复杂任务有效完成率也下降
- 说明复杂题没有进入正确路径

2. 平均材料数下降，Partial 率上升
- 说明证据厚度不足影响了完整交付

3. 升级率上升，但复杂任务有效完成率不升反降
- 说明系统虽然升级了，但 complex 链质量不够稳定

## 五、当前阶段建议展示格式

如果做一页看板，建议只展示这 4 个核心数：

- 北极星：复杂任务有效完成率
- Guardrail 1：Partial 率
- Guardrail 2：复杂任务升级率
- Guardrail 3：平均材料数

每个指标后面附一行：
- 指标定义
- 当前口径
- 风险提示

## 六、一句话总结

A1 的指标体系不该只看“是否返回了答案”，而要看：

`复杂任务是否真正完成、是否完整交付、是否走了正确路径、是否有足够材料支撑。`

## 七、v1 冻结口径（代理指标）

**版本**：`product_metrics_v1`  
**性质**：代理指标，**非**用户调研满意度，**非**线上留存/转化等经营指标。

### 7.1 复杂任务有效完成率（北极星代理）

**分子（单次 turn 视为有效完成）** — 须同时满足：

| # | 条件 | 统计字段（`extra`，由 `turn_exit_gate` 写入） |
| --- | --- | --- |
| 1 | 属于复杂任务范围 | `is_complex_task == true` |
| 2 | 任务完成 | 顶层 `task_status` ∈ `{done, succeeded}` |
| 3 | 无 insufficiency | `insufficient_evidence == false` |
| 4 | 质量门通过 | `quality_gate_passed == true` |

**分母**：同一次响应中 `is_complex_task == true` 的 turn 数。

**复杂任务范围（v1）**：`mode=complex` **或** `executor_profile∈{complex,async}` **或** `pending_kind` 含 `escalate_to_complex` / `escalate_to_async` **或** `complex_candidate=true` 且 `complex_reason_codes` 命中强信号（实现见 `complexity_policy.is_complex_task_scope`，由 `turn_exit_gate` 写入 `is_complex_task`）。

### 7.2 Guardrail 字段映射

| 指标 | 计算（v1） | 字段 |
| --- | --- | --- |
| Partial 率 | `task_status == partial` / 全部 turn | 顶层 `task_status` |
| 复杂任务升级率 | `is_complex_task` / `complex_candidate` 或 ingress 标记为 complex 的 turn | `is_complex_task` + ingress trace |
| 平均材料数 | mean(`v15_retrieved_chunks_count` + 临时材料数) | `extra.v15_retrieved_chunks_count`, `extra.v15_temporary_materials_count` |

### 7.3 数据源（v1）

| 优先级 | 来源 | 说明 |
| --- | --- | --- |
| 1 | Chat 响应 JSONL 导出 | `scripts/aggregate_product_metrics.py` |
| 2 | PG `turns` 表 | 当前仅存摘要字段；**完整 v1 需 Phase 2 存 `extra` 快照** |
| 3 | pytest 产物 | 仅 offline benchmark，不作产品看板主数据源 |

### 7.4 内测场景完成率

`用户场景与失败复盘.md` 中 62%–85% 为 **内测抽样观察**，不写入 PRD hard SLA，不作 CI 门禁。
