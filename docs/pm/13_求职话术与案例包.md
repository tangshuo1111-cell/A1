## 1. 最适合的岗位方向

基于你现在的项目经历、AI 工具使用方式和岗位要求匹配度，最适合你的不是泛化的“所有 AI 产品经理岗位”，而是下面两个岗位方向。

### 方向一：AI 自动化 / AI Workflow 产品经理

对应岗位组：
- 岗位 B
- 岗位 F

为什么最适合：
- 这类岗位重视 AI Workflow、Agentic AI、AI Coding 工具、MCP、Skills、Context Engineering
- 也重视 PRD、原型、业务逻辑、接口规则、跨部门推进和业务指标
- 你本身在项目里已经体现出：
  - 会把多来源能力组织成实际产品流程
  - 会用 AI 工具协作推进代码、测试、文档和验证
  - 对自动化、工具链、数据流和交付链有实际理解

你在这条方向上最该强调的优势：
- 不只是“会用 ChatGPT”，而是真的会用 Codex / Claude Code 类工具推进项目
- 不只是会写文档，而是能把 AI 能力组织成产品工作流
- 对 API、工具调用、状态语义、测试验证、交付链条有比普通 PM 更强的理解

适合你的简历标题写法：
- AI Workflow / Agent 产品经理
- AI 自动化产品经理
- AI Native 产品经理

> 口径提醒：讲"多 Agent"时，统一说成 **LLM 增强的分层工作流编排（complex 才走 Main/Middle/Answer 三层）**，不要说成"多个 Agent 自主对话/互相协作"。被追问时如实说明：控制流由规则主链决定，LLM 用于意图识别与最终生成。

### 方向二：AI Agent / 多 Agent / RAG 产品经理

对应岗位组：
- 岗位 B
- 岗位 G

为什么也很适合：
- 这类岗位重视 Agent 规划、意图理解、任务拆解、工具执行、结果生成、知识库集成、RAG、工具调用、多智能体协作、评估指标
- 你的项目里已经有：
  - fast / complex / async 分流
  - Main / Middle / Answer 三层协作
  - 多来源处理
  - pending / commit / retrieval 知识链路
  - 路由、状态语义、测试矩阵和评估思路

你在这条方向上最该强调的优势：
- 你不是只会讲 Agent 概念，而是做过真实主链
- 你理解多 Agent 不是“更炫的技术”，而是不同问题需要不同处理方式
- 你能讲清楚 RAG、共享能力层、复杂链升级、质量门和统一交付之间的关系

适合你的简历标题写法：
- AI Agent 产品经理
- RAG / Agent 产品经理
- 多 Agent AI 产品经理

### 不建议你优先去卷的方向

当前不建议优先把自己包装成：
- 纯策略型 C 端增长产品经理
- 纯算法训练 / 模型微调型产品经理
- 强行业经营结果导向、但你还没有足够业务实绩支撑的商业化产品经理

原因不是你不能投，而是你现在最强的证据仍然来自：
- AI 工具实操
- Workflow / Agent / RAG 产品理解
- 真实原型和验证能力

## 7. 1 分钟项目介绍

A1 是一个面向学习复盘和知识沉淀场景的 AI 产品原型。它想解决的问题不是“用户缺信息”，而是高价值信息散落在视频、网页、文档和笔记里，时间一长很容易流失，之后也很难再次调用。

我在这个项目里做的核心判断是，把“即时处理”和“长期沉淀”分开。也就是说，系统先帮用户处理当前问题，比如提取网页正文、做多来源比较、整理视频内容；如果用户确认这份内容值得留下，再明确保存到知识库，支持后续检索、回查和复盘。

当前这个项目最成立的两个价值是：第一，视频和资料可以进入长期复用链路；第二，多个外部来源可以在当轮直接比较。更长期的方向，则是从资料管理继续往长期复盘助手推进。

## 8. 3 分钟项目介绍

我做 A1 的起点，一方面是想验证多来源处理、RAG、复杂任务分流和 Agent 协作这些 AI 能力，在真实产品里到底能不能被组织起来；但真正让我持续做下去的，是一个很具体的用户问题：高价值信息虽然很多，但分散在视频、网页、文档和个人笔记里，很容易看过就丢，之后也很难真正复用。

所以我没有把这个项目做成一个泛化的聊天机器人，也没有只停留在“上传资料后问答”。我更关注两件事：第一，资料能不能在以后继续被找回来、继续提问、继续复盘；第二，用户在当下就有比较需求时，系统能不能直接处理多个来源，而不是强迫他先做知识管理。

基于这个判断，我把产品分成了两条路径。第一条是长期沉淀路径，资料先被处理成当轮可用结果，用户确认值得留下后，再保存进知识库，后续支持检索、回查和追问。第二条是即时处理路径，用户可以直接给多个网页或视频链接，系统在当轮提取、提炼和比较，不要求先入库。

这里我特别重视一个边界：A1 不会把“抓到了内容”自动等同于“应该直接入库”。我更倾向于保留人在回路，让用户先看结果、再决定保存。这样更符合知识沉淀产品的真实心智，也能避免低价值或错误内容进入知识库。

从当前阶段看，这个项目已经形成了真实可跑的原型，最成立的价值是视频复盘、资料沉淀、知识回查和多来源即时比较。更深的复杂分析和长期个人复盘，我会把它作为重点方向，但不会包装成已经完全成熟的能力。

## 9. SCQA 版

### S：背景

AI 时代信息越来越多，尤其是视频、网页、文档和个人笔记里的高价值内容，获取越来越容易。

### C：冲突

但这些信息虽然被看到、收藏甚至记录下来，却很难长期沉淀，也很难在以后再次调用。很多通用工具更擅长提取和总结，不一定能支持持续复盘。

### Q：问题

那一个 AI 产品到底应该怎么做，才能既帮助用户处理当前问题，又把高价值内容真正变成长期可复用的知识资产？

### A：回答

A1 的答案是把“即时处理”和“长期沉淀”分开：先处理当前问题，再由用户决定是否保存为知识资产；同时保留多来源当轮比较能力，避免产品退化成普通知识库。

## 10. STAR 版

### STAR 1：为什么要把即时处理和长期沉淀分开

S：
我在做这个项目时，发现很多知识工具会天然把“抓到内容”和“写入知识库”绑在一起。

T：
我要判断这个产品到底应该是自动沉淀所有内容，还是把保存动作交给用户确认。

A：
我结合自己的使用场景和当前代码现实，把产品定义成“先处理、后保存”。也就是说，先让系统回答、提取或比较；如果用户确认这份内容值得留下，再明确保存到知识库。

R：
这样产品边界更清楚，也更符合真实使用逻辑。它既避免了错误或低价值内容直接入库，也让“即时处理”和“长期复用”形成了更清晰的双路径结构。

### STAR 2：为什么不把复杂分析当成当前成熟卖点

S：
这个项目里已经有复杂链路、多来源比较和多 Agent 协作，看上去很容易讲成“已经具备复杂分析能力”。

T：
我要判断哪些能力可以作为当前成立价值，哪些只能作为方向。

A：
我去区分了已验证能力、待验证能力和长期愿景。把视频复盘、资料沉淀、知识回查和即时比较定义为当前成立能力；把更深层的长期个人复盘和高稳定复杂分析定义为重点探索方向。

R：
这样既保留了产品想象力，也避免了过度包装。后面写 PRD、项目卡和求职说明时，口径会更稳。

### STAR 3：怎么证明自己不是只会讲概念

S：
很多 AI 产品项目容易停留在“我想做什么”，但很难证明自己知道怎么验证。

T：
我要把这个项目从“产品想法”推进到“可评估的产品原型”。

A：
我补了场景复盘、指标看板、竞品分析、深度方案和路由与能力测试矩阵，把产品拆成可验证的命题，例如资料沉淀是否成立、即时比较是否成立、复杂任务是否真的该升级。

R：
这让我后面不只是能讲产品方向，还能讲“为什么这么测、看什么结果、结果出来后怎么解释和迭代”。

## 11. PREP 版

### 题目：你为什么认为这个产品不应该自动沉淀一切内容？

P：
我认为知识沉淀产品不应该默认自动保存所有处理结果。

R：
因为用户真正需要的是高价值知识资产，而不是把所有抓到的内容都堆进库里。

E：
在 A1 里，我把网页、视频、文档的处理和知识沉淀分开设计。系统先处理当前问题，用户确认内容值得留下后，再明确保存到知识库。这样更符合真实使用习惯，也能减少低价值或错误内容进入知识库。

P：
所以我把“先处理、后保存”当成 A1 的核心产品判断之一，而不是把自动入库当成理所当然。

### 题目：你为什么不把这个项目定义成普通知识库？

P：
我不把 A1 定义成普通知识库，因为它不仅解决“存内容”，还想解决“如何持续使用这些内容”。

R：
如果只做知识库存储和问答，产品很容易停留在资料管理层，难以形成即时比较和长期复盘这两个更有价值的场景。

E：
A1 当前既支持资料处理后保存入库，也支持用户直接给多个外部链接做当轮比较。这两条路径一起，才构成了它的产品差异。

P：
所以我更愿意把它定义成“从资料管理走向长期复盘助手”的产品原型。

## 12. 高频问答

### 8.1 你的产品和 NotebookLM / Coze 有什么不同？

NotebookLM 和 Coze 这类产品或平台已经能很好地完成资料上传、文本提取和基础问答。A1 的重点不在于证明“这些我也能做”，而在于两个判断：第一，是否把即时处理和长期沉淀拆开；第二，是否让产品从资料管理继续走向长期复盘。

### 8.2 你的项目当前最成立的价值是什么？

当前最成立的价值有两个：一个是视频、网页、文档和笔记进入长期复用链路；另一个是多个外部来源的当轮即时比较。更深层的长期个人复盘和复杂分析，我会把它定义成方向，而不是当前成熟卖点。

### 8.3 你怎么证明自己不是只会写文档？

我这个项目不是停留在方案层，而是有真实可跑的后端主链。我也补了测试矩阵、sandbox 样本、指标看板和场景复盘，所以我能讲的不只是“我想做什么”，还包括“系统现在做到什么、哪里没做到、为什么没做到、下一步怎么验证”。

### 8.4 你的北极星指标是什么？达标了吗？

我有两个北极星：北极星 1 是"资料二次调用率"，**已接线最小埋点**（commit 打 `source_kind`、检索命中落 metrics）；沙箱代表题未全覆盖 commit→检索 时样本不足，只看趋势。北极星 2 是"复杂任务有效完成率"，已经接线（出口写入 `turn_product_metrics`，周报三态判定）。

关于"达标没"，我会诚实区分：

> 北极星 2 在代表题沙箱里约 **58.6%（17/29 complex，2026-06-22 真实 LLM 复跑）**，目标线我设的是 70%；但当前复杂子集 N < 我设的有效下限 30，所以周报判定是「**样本不足**」——不是"已达标"，也不是"未达标"。**这是观测线，不是线上门禁**。工程 release 看的是 42/42 回归 + 7/7 真实能力 smoke（**均为 staging / 本地真实 LLM，`LIGHT_MAQA_FAKE_LLM=0`；非默认 push CI**），产品价值看北极星趋势加 staging 人工验收。

我特意没有在小样本上设 pass/fail，因为复杂子集还没到统计有效区间（我给的下限是 ≥30）之前，硬说"已达标"反而会误导。**知道什么时候不能外推，是这个指标设计里我最在意的一点。**

> 口径提醒：不要假装"线上已跑通北极星 SLO"。终面被追问时，承认"埋点在、数据是离线代表题、样本小所以只看趋势"比硬编"生产已达标"更稳，也更像 PM。

## 13. 面试时最该记住的 5 句话

1. A1 的核心不是一次性提取信息，而是把高价值内容变成长期可复用的知识资产。
2. 我把即时处理和长期沉淀分开，先回答、后保存，而不是默认自动入库。
3. 这个项目不是在堆能力，而是在验证双路径产品主线是否成立。
4. 当前最成立的价值是视频复盘、资料沉淀、知识回查和多来源即时比较。
5. 长期个人复盘和更深层复杂分析是重点方向，但我不会把愿景讲成现实。
