# MVP 与优先级

目标：
从“什么都能做”收敛到“什么必须先做”。

## MVP 定义

MVP 不是功能越少越好，而是最小可验证闭环。对 A1 来说，MVP 不是把所有材料能力都塞进去，而是保留最能证明产品价值的最小链路。

A1 的 MVP 建议：
- 一个统一主入口
- 一条稳定默认主路径
- 三类核心材料：知识库、文档、网页
- 一个明确的复杂问题升级机制
- 一个统一结果出口
- 一组不会回退的核心验证指标

## 需求优先级原则

评估维度：
- 用户价值
- 对主链稳定性的影响
- 实现复杂度
- 验证效率
- 对转岗作品表达的帮助

## P0：必须先做

- 统一入口路由
- 单一事实源
- 统一材料流语义
- 统一质量门与出口门
- 默认路径真实可跑
- 核心测试与 benchmark 不回退

原因：
这些能力决定项目是不是“产品系统”，不解决就只是能力堆叠。

## P1：应该做

- 文档、网页、知识库三类主材料链路稳定
- 异步任务状态反馈
- 复杂问题升级规则更清晰
- 前端对任务状态和材料依据展示更友好

原因：
这些能力决定用户能不能真正理解和使用系统。

## P2：可以延后

- 视频处理深度优化
- 更细的多 Agent 协作策略
- 更多外部能力接入
- 更复杂的运营和配置后台

原因：
这些能力会增强系统，但不是 MVP 成立的前提。

## P3：当前不优先

- 商业化计费系统
- 多租户权限体系
- 企业级管理后台
- 大规模自动化运营能力

原因：
当前阶段主要目标是验证产品主链，而不是做商业化全家桶。

## 推荐路线

第一阶段：
做稳定主链

重点解决：
- 路由收敛
- 材料流统一
- 单一事实源
- 出口统一

第二阶段：
做用户可感知体验

重点解决：
- 材料依据展示
- 状态反馈
- 异步体验
- 复杂问题交付质量

第三阶段：
做差异化强化

重点解决：
- 视频与多模态增强
- 更强的复杂任务处理
- 更细的产品指标体系

一句话总结：
A1 的 MVP 不是“支持最多能力”，而是“用最少但稳定的能力，证明复杂信息任务可以被统一主链稳定处理”。
