## 1. 这份文档要回答什么

这份文档不再重复讲“产品是什么”，而是重点回答五个问题：

1. 为什么这个产品值得做
2. 为什么我选择当前这套方案，而不是做成另一个通用知识库或插件平台
3. 这个产品当前最应该验证什么
4. 我准备怎么测试，为什么这么测
5. 测试结果应该沉淀在哪里，后续如何基于结果迭代

这份文档对应的内容清单主要包括：
- 大模型在这个场景中的不可替代性

- 效果保障
- 测试、灰度、A/B 与后续维护

## 2. 为什么要做这个产品

我做这个产品，不是因为“AI 什么都能做”，而是因为我认为下面这个问题真实存在，而且越来越重要：

`高价值信息越来越多，但视频、网页、文档和笔记里的内容，很难被长期沉淀、再次调用和持续复盘。`

这不是一个纯工具问题，而是一个产品问题。

如果只做收藏、搜索、标签，用户能把内容存下来，但未必能在以后继续使用。  
如果只做聊天问答，系统也许能回复，但未必能把这些内容变成长期可复用的知识资产。

所以我想验证的是：

`AI 能不能帮助用户，把一次性消费的信息，变成长期可复用、可回查、可继续复盘的知识资产。`

但这里有一个前提判断：不是所有被抓到的内容都应该立刻沉淀。A1 当前更强调“先处理、后保存”，即先把内容处理成当轮可用结果，再由用户确认是否进入长期知识库。

## 3. 为什么选择当前方案

### 3.1 没有把它做成“又一个通用知识库”

原因很直接：上传资料、提取文本、做基础问答，这些能力今天已经越来越商品化。

如果我只停留在这里，这个产品的差异化会很弱。

### 3.2 没有把它做成“又一个平台型 Agent 工具”

像 Coze、Dify 这类平台很强，但它们主要解决的是“快速搭应用”的问题。  
我当前更想验证的是一个具体产品命题，而不是做平台。

### 3.3 为什么是“双路径”

我最终选择的不是单一路径，而是两条互补路径：

1. 长期沉淀路径
   - 解决“以后还能不能找回来、继续问、继续复盘”
   - 这条路径不是自动入库，而是先处理，再由用户决定是否保存
2. 即时比较路径
   - 解决“我现在就想对多个来源做比较，不想先做资料管理”

为什么这么选：
- 如果只有沉淀路径，产品容易退化成“又一个知识库”
- 如果只有即时比较路径，产品很难形成长期价值闭环

所以这两条路径一起，才更接近我真正想验证的产品方向。

这也意味着，A1 不是“所有内容先进库再说”的产品。它把即时处理和长期复用拆开：当轮先完成问题处理，只有用户明确认可这份内容值得留下，才进入知识沉淀链路。

### 3.4 当前平台和“复盘 / 知识积累”的关系

从现有市场看：
- 平台型产品解决的是“能力搭建”问题
- source-grounded 产品解决的是“围绕给定资料问答和整理”问题

但“长期复盘”和“知识积累”要求更高，它不仅要求把内容导入系统，还要求：
- 以后还能找回
- 找回以后还能继续问
- 多个来源之间能形成可用判断

这就是 A1 想补的空白：不是只把资料收进来，而是让资料之后继续产生价值。

## 4. 为什么这个场景需要大模型

### 4.1 传统方案能解决什么

传统收藏、搜索、标签和文件夹能够解决：
- 把内容保存下来
- 按关键词或标题查找

但它们不擅长：
- 把视频、网页、文档、笔记拉到同一语义层理解
- 围绕材料做摘要、比较和追问
- 基于已有资料生成结构化、可交付的结果

### 4.2 大模型的不可替代性

在 A1 这个场景里，大模型更重要的价值不是“能多说几句话”，而是：

1. 统一理解异构材料
2. 组织多来源内容
3. 在已有材料基础上进行比较、归纳和交付

也就是说，大模型在这里承担的是“知识处理能力”，而不是单纯的聊天能力。

### 4.3 A1 对大模型的目标

当前阶段，大模型目标不是“替代所有逻辑”，而是：
- 提升材料理解和组织能力
- 提升多来源比较和复杂任务交付能力
- 在材料不足时更诚实、更保守，而不是硬答
## 5. 当前方案为什么可行

我判断这套方案可行，主要基于三个现实条件：

1. 真实需求已经足够明确
   - 视频复盘、资料回查、多来源比较都对应很具体的用户任务
2. 当前代码里已经有两条路径基础
   - 一条偏长期沉淀，一条偏即时比较
3. 当前系统已经具备分流、补材、统一交付的主链基础

所以现在不是从 0 想象一个产品，而是在已有主链上验证“哪条价值最成立”。

## 6. 模型、Prompt 与数据标准

### 6.1 大模型选型思路

当前更重要的不是在文档里写“最强模型”，而是按任务拆职责：
- 路由判断更强调稳定性和边界感
- 材料组织和比较更强调结构化能力
- 最终交付更强调表达克制和基于材料

这也意味着，模型选择不是一个统一答案，而是要服务于不同阶段目标。

### 6.2 Prompt 工程写在哪里

当前 prompt 资产并不是一套统一提示词，而是按角色分层：
- MainAgent prompt：负责协作总判断
- MiddleAgent prompt：负责材料裁决
- AnswerAgent prompt：负责最终表达策略
- Answer 内部执行器 prompt：负责最终文本生成
- Fast 路径还有独立轻量 prompt

代码位置分别在：
- [backend/agents/main_agent/prompt.py](/D:/1/A1_publish/backend/agents/main_agent/prompt.py)
- [backend/agents/middle_agent/prompt.py](/D:/1/A1_publish/backend/agents/middle_agent/prompt.py)
- [backend/agents/answer_agent/prompt.py](/D:/1/A1_publish/backend/agents/answer_agent/prompt.py)

### 6.3 数据 / 材料构建标准

当前更贴合代码现实的，不是“训练数据集”这个词，而是：
- 知识材料构建标准
- 评测样本构建标准

现阶段重点数据包括：
- 视频字幕 / 转写文本
- 网页正文
- 文档解析文本
- 用户笔记
- 比较任务中的当轮临时材料

标准重点在于：
- 可被后续检索和引用
- 尽量保留来源信息
- 能支持后续比较、追问和复盘

### 6.4 执行流程与后续维护

执行流程不是“训练一个模型再上线”，而是：
- 材料进入系统
- 系统判断走哪条路径
- 复杂任务在必要时升级
- 最终通过统一出口交付

后续维护重点则是：
- 路由是否稳定
- 材料质量是否稳定
- 复杂任务交付是否回退
- 指标和样本集是否持续更新

## 7. 当前最该验证的不是全部能力，而是三个命题

### 命题 1：资料沉淀是否真的成立

要验证的不是“能不能上传”，而是：
- 视频、网页、文档进入系统后，是否真的先变成了当轮可用材料
- 用户在看过结果后，是否愿意明确保存
- 用户是否能在后续围绕这些材料继续回查和追问

### 命题 2：即时多来源比较是否真的成立

要验证的不是“系统有没有抓取到网页”，而是：
- 多个链接是否能在当轮被稳定提取
- 是否能形成基本可用的比较结果
- 是否不是简单摘要拼接

### 命题 3：复杂链路是否比普通直答更有价值

要验证的不是“系统有没有 complex 路径”，而是：
- 哪些问题确实应该升级
- 升级后是否真的改善了结果
- 复杂链路是否有稳定交付，而不是只是更慢

## 8. 准备怎么测试

我现在更适合采用“三层验证”。

### 5.1 第一层：路由与路径验证

要测什么：
- 什么问题走 Fast
- 什么问题走复杂链路
- 什么问题进入异步处理
- 什么问题应该保守返回，而不是硬答

为什么先测这层：
- 因为这个产品不是单一路径产品
- 如果路径本身选错，后面谈回答质量没有意义

建议测试方式：
- 设计按路径分类的固定样本集
- 每条样本明确预期：应该走哪条路、最终状态应该是什么

### 5.2 第二层：能力场景验证

要测什么：
- 视频转文字沉淀并复盘
- 网页 / 文档 / 笔记沉淀后长期回查
- 多链接即时比较
- 复杂问题升级后是否真的更有价值

为什么要测这层：
- 路由正确，不代表用户价值成立
- 这层是为了验证“这条能力到底有没有形成产品价值”

建议测试方式：
- 每个核心场景至少保留 1 组稳定样本
- 复杂场景单独保留专项测试集，而不是混在一起

### 5.3 第三层：交付语义验证

要测什么：
- 最终到底是 completed、partial 还是 pending
- insufficiency 是否合理
- 复杂任务是否真的完成，而不是只返回一段像答案的话

为什么要测这层：
- AI 产品常见问题不是没返回，而是“看起来返回了，其实没完成”
- 这正是你这个项目很重视的部分

建议测试方式：
- 针对最终状态字段和交付语义做断言
- 不只看回答文本

## 9. 为什么要这样测

因为这个产品的风险不在“功能不存在”，而在三个地方：

1. 路径选错
   - 复杂问题被简单处理
   - 简单问题被过度升级

2. 能力成立但价值不成立
   - 例如资料虽然入了系统，但以后没人再用
   - 或者比较虽然做了，但没有形成真正有价值的差异提炼

3. 结果语义不稳定
   - 系统内部已经知道材料不足，但用户看到的像“已经完成”

所以这套测试设计不是为了证明“功能能跑”，而是为了证明：

`这条产品主线到底有没有成立。`

## 13. 灰度、A/B 与 shadow 的位置

### 13.1 当前阶段优先级

当前更适合的顺序是：
1. 离线样本集
2. 路由覆盖和 sandbox 回归
3. shadow 对比
4. 小流量灰度
5. 条件成熟后再做 A/B

### 13.2 为什么不是一开始就做 A/B

因为在路由和状态语义都还没完全稳定前，直接做 A/B 很容易把问题混在一起。

当前阶段更重要的是先证明：
- 这套路径判断本身稳定
- 复杂链路交付逻辑稳定
- 结果语义稳定

### 13.3 如果以后做灰度 / A-B，应该测什么

可以重点看：
- 复杂任务有效完成率
- Partial 率
- 复杂任务升级率
- 平均材料数
- insufficiency 占比

## 14. 做完测试后，要怎么解释结果

测试结果不应该只停留在“通过 / 不通过”，而要回答：

### 9.1 如果路由错了

说明问题主要在：
- 路径判断逻辑
- 复杂任务识别规则

下一步动作：
- 优先调路由与升级逻辑

### 9.2 如果路由对了，但结果还是浅

说明问题主要在：
- 复杂链路没有真正发挥作用
- 材料组织和比较深度不够

下一步动作：
- 优先看复杂链路的材料组织、比较和交付方式

### 9.3 如果回答看起来有内容，但状态语义不稳

说明问题主要在：
- 最终交付定义不清
- partial / insufficiency / pending 判断不稳定

下一步动作：
- 优先统一结果语义，而不是先调回答文案

### 9.4 如果沉淀成立，但没人再次使用

说明问题主要在：
- 产品闭环没有真正形成
- 资料进入系统，但后续复用价值弱

下一步动作：
- 优先优化回查、追问和复盘体验

## 15. 后续打算怎么做

基于当前产品阶段，我更建议按下面顺序推进：

1. 先把路由覆盖样本集补齐
2. 再把核心场景样本集固定下来
3. 再把结果挂进现有 sandbox / 周报 / 指标脚本
4. 跑出第一轮稳定结果
5. 基于结果决定先优化：
   - 路由
   - 复杂链路
   - 材料质量
   - 最终状态语义

## 16. 结论

这份深度方案真正想说明的，不是“系统理论上能做什么”，而是：

`我已经把这个产品拆成了几个可验证命题，并且明确了为什么这么测、测什么、结果去哪看、测完后怎么解释和迭代。`

这比单纯写“我用了 RAG、Agent、Prompt、多来源处理”更像 AI 产品经理真正会做的事。
