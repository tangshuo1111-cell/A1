# 20 KB补强：主链边界与复杂题升级规则

## 目的

本文件是为 `KBX2`、`KBX4`、`KBX5`、`KBX8`、`KBX9`、`KBX10` 定向补强的结构化知识材料。

目标不是描述所有实现细节，而是给知识库提供一套足够稳定、可直接命中的判断口径：

- 为什么复杂题没有稳定进入 Agent 主链
- Agent 主链、capability 主链、async task 主链分别适合什么
- 复杂题应该如何从 fast 升级到 Agent
- 当前系统中哪些问题是架构协作问题，哪些已经降级成实现收口问题

## 一、三条主链的职责边界

### 1. Agent 主链

Agent 主链负责：

- 复杂分析
- 多维判断
- 比较与取舍
- 正反论证
- 基于多份材料做综合结论
- 最终回答组织与审慎表达

Agent 主链最适合解决的问题：

- 复杂题
- `complex candidate`
- 需要 Planner / Main → Middle → Answer 协作的问题
- 需要“先收集材料，再组织判断”的问题

Agent 主链最不适合解决的问题：

- 长后台重处理
- 视频下载、切段、ASR、OCR 这类工程型重任务
- 明确只有单一事实、单一术语解释、单资料总结的问题

### 2. capability 主链

capability 主链负责：

- 视频
- 网页
- 文档
- 知识检索
- pending / commit 等专项能力

capability 主链最适合解决的问题：

- 工具型任务
- 单资料快处理
- 重处理前置准备
- 把材料变成 Agent 可消费的临时材料、`pending material` 或已提交知识

capability 主链最不适合解决的问题：

- 多步推理
- 深比较
- 复杂取舍
- 最终审慎表达

### 3. async task 主链

async task 主链负责：

- 长视频
- 大文档
- 明确超出同步预算的重任务
- 需要后台 worker、任务状态与轮询的处理

async task 主链最适合解决的问题：

- 处理成本高
- 同步预算内不可能完成
- 适合先给 pending / task，再后台完成

async task 主链最不适合解决的问题：

- 本可在同步内完成的复杂判断题
- 需要即时给用户做取舍、排序、路线图的知识题

## 二、为什么复杂题没有稳定进入 Agent 主链

当前系统里“复杂题没有稳定进入 Agent 主链”的原因，不是单一 bug，而是三类因素叠加：

### 1. 路由与升级门槛曾经偏宽

系统默认先 `fast` 是合理的，但如果 `quality_gate` 对复杂题过于宽松，就会出现：

- 复杂题被 `fast` 先答
- 答案看起来“勉强可用”
- 但其实没有真正完成比较、取舍、分情况分析
- 最终没有升级进 Agent 主链

所以问题不是“没有 complex candidate”，而是：

- `complex candidate`
- `fast -> Agent`
- `quality_gate`

这三者曾经没有形成足够强的升级闭环。

### 2. 材料流与质量门控曾经没有完全统一

如果：

- shared retrieval
- `kb_sufficiency`
- `quality_gate`
- `feedback_gate`
- `approval_gate`

不是按统一协议工作，就会出现：

- fast 看的是一份材料
- Middle 又自己重新判断一份材料
- complex 再走另一套自治补料

这样复杂题即使进入了 Agent，也可能没有稳定使用同一份证据集。

### 3. 复杂题进入 Agent 不等于复杂题已经完整交付

当前要区分两件事：

1. 复杂题能否稳定升级进 Agent 主链
2. 复杂题进入 Agent 后，能否完整、高质量交付

第一件事解决后，第二件事仍可能存在：

- partial
- 预算内只能给 round0
- 材料不足
- 结构完整但证据不够厚

因此“没稳定进入 Agent”与“进入 Agent 但答得不够完整”是两个不同问题。

## 三、complex candidate 的稳定判定

当前更合理的口径是：

### 默认视为 `complex candidate` 的任务形态

- 比较类
- 取舍 / 决策类
- 多维分析类
- 正反论证类
- 跨材料综合类
- 方案设计 / 路线图类

### 默认不视为 `complex candidate` 的任务形态

- 单事实问答
- 单术语解释
- 单资料总结
- 低推理负担的简单列点

关键点：

- 视频题不天然复杂
- KB 题不天然复杂
- 是否复杂，按任务形态判，不按资料类型判

## 四、fast 为什么不能吞掉复杂题

对 `complex candidate`，更稳的规则应是：

- 默认先 `fast`
- 但 `fast` 只要不明显优秀，就应升级到 Agent

这里“不明显优秀”至少包括：

- 没完成用户要求的维度
- 没有真正做比较、取舍或判断
- 只是压缩复述
- 证据不足
- 结构浅但看起来像答完了

这条规则的意义在于：

- 默认先快
- 但复杂题不能只因为“表面有答案”就停在 `fast`

## 五、当前系统里哪些问题已经从架构问题降级成实现收口问题

已经明显从架构协作问题降级为实现收口问题的：

- `multisource_compare` 并入统一 `quality_gate` 二轮模型
- `approval_gate` 与 commit 闭环
- `material trace` 全路径输出
- shared retrieval / `kb_sufficiency` / `quality_gate` 协议统一

这些问题的核心设计已明确，后续主要是：

- 代码一致性
- trace 一致性
- baseline 与文档收口

仍然不是单纯实现收口问题、而是能力问题的：

- 复杂题升级阈值是否足够稳
- complex 题进入 Agent 后是否能完整交付
- KB 检索证据的覆盖度与厚度是否足够

## 六、直接结论

如果要解释“当前系统为什么会出现复杂题没有稳定进入 Agent 主链”，最准确的总结是：

1. 系统默认先 `fast` 的方向是对的
2. 真正的问题在于 `quality_gate` 曾经对复杂题偏宽
3. 复杂题升级到 Agent 的标准，需要比“有个可用答案”更严格
4. 复杂题进入 Agent 主链后，还要继续解决完整交付与证据厚度问题

换句话说：

**当前系统的问题，不再是“有没有 Agent 主链”，而是“复杂题升级是否足够稳定，以及进入 Agent 后能否高质量交付”。**
