# 02 环境变量与数据库

本文档与根目录 `docker-compose.yml`、`.env.example` 对齐；生产环境请修改口令与主机。

## PostgreSQL（Docker Compose 服务名：`postgres`）

| 项 | 值 |
| --- | --- |
| 镜像 | `postgres:16-alpine` |
| 服务名 | `postgres` |
| 容器内端口 | `5432`（通常映射到主机 `5432`） |
| `POSTGRES_USER` | `light_maqa` |
| `POSTGRES_PASSWORD` | `light_maqa_dev` |
| `POSTGRES_DB` | `light_maqa` |
| 健康检查 | `pg_isready -U light_maqa -d light_maqa` |
| 数据持久化 | Docker 卷 `pg_data`（**勿**与 `_local/data` 下旧 SQLite 混淆） |

## `DATABASE_URL`

语法（libpq 兼容）：

```text
postgresql://用户名:密码@主机:端口/数据库名
```

**本机**直接用 Python / `pytest` / `scripts/run_dev.py` 连接运行在 Docker 内的 PostgreSQL（主机用 `127.0.0.1` 或 `localhost`）：

```text
postgresql://light_maqa:light_maqa_dev@127.0.0.1:5432/light_maqa
```

**Compose 网络内**（例如 compose 中的 `api` 服务）连接同一套账号时，主机名必须使用服务名 `postgres`：

```text
postgresql://light_maqa:light_maqa_dev@postgres:5432/light_maqa
```

二者不要混用：`127.0.0.1` 仅在本机进程访问容器映射端口时成立；容器互访用 `postgres`。

可选连接参数示例：`?connect_timeout=10`（若 URL 已包含，以应用侧实现为准，避免重复冲突）。

## `.env` 与 `.env.example`

- **`.env.example`**：模板，可提交，可进入分析包 / 交付包。
- **`.env`**：本地真实密钥与连接串，**不提交**、**不打包**。

若无 `.env`，可在仓库根执行：

```powershell
Copy-Item .env.example .env
```

再在 `.env` 中填写例如：

- `DATABASE_URL=postgresql://light_maqa:light_maqa_dev@127.0.0.1:5432/light_maqa`
- `API_BEARER_TOKEN=...`（按需）
- 各类 LLM / 外部服务 Key（按需）

其他运行项（`APP_ENV`、`LOG_LEVEL`、`DATA_DIR` 等）以 `.env.example` 为准。

## 检索相关环境变量

当前统一口径：

- 对业务方默认只承认 `RETRIEVAL_MODE=auto`
- `keyword / semantic / hybrid` 仅作为内部调试/验收策略保留

推荐基线：

```text
RETRIEVAL_MODE=auto
EMBEDDING_ENABLED=1
RAG_HYBRID=1
CHAT_SYNC_BUDGET_MS=30000
```

含义说明：

- `RETRIEVAL_MODE=auto`
  - 统一检索入口默认只请求 `auto`
  - 真正落到 `keyword / hybrid / semantic` 由运行时自动决定
- `EMBEDDING_ENABLED=1`
  - **唯一语义检索开关**（同时管「查」与「存」）：
    - **查**：`auto` 会优先尝试 `hybrid`；若 `rag_embeddings` 无数据或语义链异常，回退 `keyword`
    - **存**：知识库 `commit` 后自动写入 `rag_embeddings`（与检索共用本开关；已废弃独立 flag `ENABLE_EMBED_ON_COMMIT`）
- `RAG_HYBRID=1`
  - 这是历史兼容变量名
  - 现在只表示 **keyword 路径内是否启用轻量 TF-IDF 重排**
  - **不表示**“默认主路就是 hybrid”
- `CHAT_SYNC_BUDGET_MS=30000`
  - 默认复杂题同步预算 30 秒
  - 超过后应优先交付 partial、缩范围或转 `async`

若要显式改回轻量环境，可至少设置：

```text
RETRIEVAL_MODE=auto
EMBEDDING_ENABLED=0
```

此时 `auto` 会稳定回退到 `keyword`。当前推荐默认态则是 **`EMBEDDING_ENABLED=1`**。

### 语义检索单一开关（`EMBEDDING_ENABLED`）

| 值 | commit 入库 | 检索 `auto` | `/health` embedding.mode |
| --- | --- | --- | --- |
| `1` | 自动写 `rag_embeddings` | 有向量 → hybrid；无向量/失败 → keyword | `index_present` 或 `enabled_no_rows` |
| `0` | 不写向量 | 稳定 `keyword` | `disabled` |

启动时若 `EMBEDDING_ENABLED=1` 且 `rag_embeddings` 无行，日志会 **warn**（避免 silent 降级）。

**已废弃**：`ENABLE_EMBED_ON_COMMIT`（勿再配置；行为由 `EMBEDDING_ENABLED` 派生）。

历史库补向量（v1 同步）：

```powershell
python scripts/reindex_embeddings.py
```

## Docker 启动与中文目录下的 project name

根目录除 `Dockerfile` 外还有 `docker-compose.yml`（条目不多，保持根目录）。

若仓库文件夹名包含中文（例如 **`项目代码`**），为避免 Compose 项目名含非法字符，请**二者择一**：

```powershell
docker compose -p rag_agent up -d postgres redis
docker compose -p rag_agent ps
docker compose -p rag_agent logs postgres --tail=50
```

或在仓库根 `.env` 中设置：

```text
COMPOSE_PROJECT_NAME=rag_agent
```

若日后根目录重命名为英文名（例如 `rag_agent_project`），仅更新你本地路径与文档中的路径示例，**不改变**业务结构；上述 `-p rag_agent` / `COMPOSE_PROJECT_NAME` 仍可沿用。

## CI 提示

CI（如 GitHub Actions）可能注入与本仓库 compose 账号一致的 `DATABASE_URL` 并启动 `postgres` 服务容器；细节以 `.github/workflows/ci.yml` 为准。

## 附录：旧 SQLite 清理

当前业务数据以 **PostgreSQL** 为准。

若在 `_local/data/` 下仍存在 `knowledge.sqlite`、`conversation.sqlite`、`runtime.sqlite`：

1. 先停止本地后端。
2. 确认没有仅此一份的重要数据；需要则先备份。
3. 再删除或移动到例如 `_local/data/backup_sqlite/`。

补充口径：

- `data/` 是可提交样本区，不应再保留这些 sqlite
- 若历史上曾误放在 `data/`，应移动到 `_local/data/backup_sqlite/`

**不要**把 SQLite 文件放入可提交的 `data/`；**不要**把 SQLite 当作当前主链数据库；**不要**与 PostgreSQL 主路径混用。

## 附录：`PLAYWRIGHT_CDN` 等（按需）
