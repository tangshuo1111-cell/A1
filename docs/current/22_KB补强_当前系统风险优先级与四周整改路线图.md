# 22 KB补强：当前系统风险优先级与四周整改路线图

## 目的

本文件面向 `KBX1`、`KBX3`、`KBX5`、`KBX8`、`KBX9`、`KBX10`，用于给知识库提供：

- 当前系统最影响真实用户体验的问题排序
- 当前最大质量风险判断
- 如果只能优先修一个方向，该修什么
- 未来 4 周整改路线图

## 一、当前最影响真实用户体验的三个问题

结合协作施工计划、项目评估、测试质量审计和评测基准，当前最影响真实用户体验的三个问题可以排序为：

### 第 1 位：复杂题升级与完整交付不稳定

表现：

- 一部分复杂题没有稳定进入 Agent 主链
- 进入 Agent 后，又可能只得到 `partial`
- 用户会感觉系统“有时很聪明，有时又像只做了半步”

这是最影响体感的问题，因为它直接作用在：

- 复杂问答
- 多维分析
- 路线图与取舍题

### 第 2 位：KB 证据覆盖度和厚度不足

表现：

- KB 主路更统一了
- 但很多题仍然只能拿到 `0/1` chunk
- 一部分题仍要退到 web fallback

这会导致：

- 答案看起来有结构
- 但证据基础不够厚
- 复杂知识题不够稳

### 第 3 位：专项重处理链的同步交付质量仍不稳定

表现：

- 重视频 / 重资料链已经接线较完整
- 但复杂或长耗时路径仍可能给出 partial
- 用户可能感受到“系统流程通了，但完整度还不稳定”

## 二、当前最大质量风险到底是什么

如果必须在：

- 路由错误
- 回答截断
- 链路不闭环

三者里选一个当前最大质量风险，我的判断是：

## 当前最大质量风险：**复杂题路由与升级不稳定**

原因：

### 1. 链路不闭环已明显改善

相比之前：

- `multisource_compare` 已并入统一二轮模型
- `approval_gate` 与 commit 闭环更清楚
- `material trace` 全路径外显

所以“链路不闭环”仍是风险，但已经不再是第一风险。

### 2. 截断是表层结果，不是唯一源头

`partial`、截断、未完整交付，常常是更深层问题的结果：

- 复杂题没有升级
- 或升级后证据不够
- 或复杂题预算与表达能力不匹配

因此“回答截断”更像现象层。

### 3. 路由与升级不稳定会放大其他问题

如果复杂题本该：

- 进入 Agent
- 或进入 complex

却被留在 fast，系统就会同时表现出：

- 答案偏浅
- 结构不稳
- 复杂题质量波动

所以从全局影响看，路由与升级不稳定仍是最大质量风险。

## 三、如果现在只能优先修一个方向，应该修什么

候选方向：

- 视频链
- KB 检索链
- 复杂题进入 Agent 主链的路由

我的推荐是：

## 优先修：复杂题进入 Agent 主链的路由与升级质量门控

理由：

### 1. 它是其他问题的放大器

如果复杂题本该进 Agent 却没进：

- KB 再强，答案也可能仍偏浅
- 视频链再通，复杂总结也可能组织不够

### 2. 它直接决定复杂题体感

用户最敏感的不是：

- 内部有几条链

而是：

- 复杂题到底有没有答到位

### 3. 它能放大已有建设的价值

你已经做了：

- 主链接线
- shared retrieval
- `quality_gate`
- `approval_gate`
- material trace

如果路由与升级门槛不稳，这些投入的价值释放不出来。

## 四、哪些问题已经从架构问题降级成实现收口问题

已经降级为实现收口问题的：

- `multisource_compare` 是否并入统一二轮 gate
- approval 与 commit 是否形成 turn 级闭环
- material trace 是否全路径输出
- shared retrieval、`kb_sufficiency`、`quality_gate` 是否按同一协议工作

这些问题的核心设计已经明确，剩余更多是：

- 一致性
- 测试
- baseline
- 边缘路径收口

仍然不是单纯收口问题的：

- 复杂题升级阈值
- KB 证据厚度
- complex 题完整交付质量

## 五、未来 4 周整改路线图

如果按“统一口径、行为收口、数据能力、专项链优化”四段来排，我建议：

### 第 1 周：统一口径

- 冻结 fast / complex / async executor profile
- 冻结 `complex candidate` 规则
- 冻结 `quality_gate`、`feedback_gate`、`approval_gate` 职责
- 冻结 temporary / pending / committed 材料层语义

目标：

- 先只有一套对外和对内都一致的说法

### 第 2 周：行为收口

- 复杂题升级路径全部走统一 `quality_gate`
- `multisource_compare`、general fast、kb_fast 不再有核心豁免
- approval 与 commit 闭环统一
- trace / baseline / integration tests 对齐

目标：

- 让系统真正只有一套运行真相

### 第 3 周：数据能力

- 检索主路统一为 `RETRIEVAL_MODE=auto` + `EMBEDDING_ENABLED=1`
- 清查 KB 覆盖度
- 对 benchmark 定向补强知识材料
- 观察 chunk 厚度能否从 `0/1` 提升到 `2~4`

目标：

- 让 KB 不只是“能命中”，而是“有足够证据厚度”

### 第 4 周：专项链优化

- complex 题完整交付质量提升
- 视频 / 文档等专项链与 Agent 的协作继续收口
- 对 partial / async 出口做更一致的用户表达

目标：

- 把“主链接上了”进一步推进成“交付质量稳定”

## 六、直接结论

如果要给当前系统一个面向架构负责人的最简判断：

1. 现在最大风险已不是“有没有架构”，而是“复杂题升级与完整交付是否稳定”
2. 如果只能优先修一个方向，应先修复杂题进入 Agent 主链的路由与质量门控
3. KB 与专项链的提升都很重要，但更像第二层放大器

一句话总结：

**当前系统最该优先解决的，是“复杂题该不该升级、升级后能不能完整交付”这一主问题；只有主问题稳定，KB 与专项链的价值才会被真正放大出来。**
