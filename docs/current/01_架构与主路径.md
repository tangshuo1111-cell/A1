# 01 架构与主路径

## 系统定位

这是一个有状态的多 Agent / RAG / Tool 编排系统，不是无状态聊天壳子。

当前现状仍以单轮主链为主要运行入口，但项目的**冻结目标架构**已经切换为：

- 外层平台图
- 内层三强自治核心

冻结口径见 [04_目标架构冻结与术语表](/D:/1/Rag_and_Agent_repo_staging/项目代码/docs/current/04_目标架构冻结与术语表.md)。

主流程：

用户请求  
→ `POST /chat/agno`  
→ MainAgent 判断  
→ MiddleAgent 收集材料 / 工具执行 / pending / commit / retrieve  
→ AnswerAgent 生成最终回答  
→ `answer + extra + trace`

## 架构分层

### 接口层

- `backend/api/`

### 聊天主链编排

- `backend/application/chat/`（单轮编排入口）
- `backend/services/`（如 `agno_chat_service` 对外装配）

### Agent 层

- `backend/agents/main_agent/`
- `backend/agents/middle_agent/`
- `backend/agents/answer_agent/`

### 知识层

- `backend/knowledge/`

### 工具与检索

- `backend/tools/`
- `backend/rag/`、`backend/retrieval/`

### 存储层

- `backend/storage/`

## 强 Agent 原则

可以拆文件，但不能把主判断权拆出 agent。

也就是说：

- `runtime` 可以拆成多个子模块
- `service` 可以 facade 化
- `workflow` 可以编排

但最终判断必须仍由各自 agent 的主入口签字。

## 目标终局说明

当前 `POST /chat/agno -> Main -> Middle -> Answer` 仍是现行主入口事实，但它不再代表项目的最终终局结构。

项目后续迁移必须回扣：

- 外层：`Semantic Router + Lane + Shared Capability Plane + Async Control Plane`
- 内层：`MainAgent + MiddleAgent + AnswerAgent`

也就是说：

- 现行主链是当前运行事实
- 平台图 + 三强自治核心是冻结后的目标事实
- 后续 Phase 2 及之后的目录与运行时工作，都必须围绕这个目标推进
