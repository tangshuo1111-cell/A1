# 17 默认路由、材料流与质量门控规则

## 状态

本文件用于冻结当前项目在以下三个问题上的第一版统一口径：

- 默认路由规则
- 材料流规则
- 质量门控与协作升级规则

本文件只冻结**已达成共识的原则**与**第一版建议阈值**。

以下内容仍视为待后续实验校准的参数：

- `fast -> Agent` 的精确阈值
- `Agent 单轮 -> 第二轮` 的精确阈值
- `KB` 证据充分度的精确数值门槛
- 少数边界题型的默认归类

## 总体原则

### 原则 1：默认先 `fast`

系统默认优先尝试 `fast`，不为“显得高级”而默认进入 Agent 主链。

### 原则 2：升级基于质量门控

`fast`、`Agent 单轮`、`Agent 第二轮` 之间的切换，不以静态想象为准，而以质量门控结果为准。

当前默认运行值：

- `RETRIEVAL_MODE=auto`
- `EMBEDDING_ENABLED=1`
- `CHAT_SYNC_BUDGET_MS=30000`

也就是说，当前系统默认就是：

- 共享检索优先尝试 `auto -> hybrid`
- 若无向量数据或语义链异常，再自动回退到 `keyword`
- 复杂题同步预算默认 30 秒，而不是 20 秒

### 原则 3：材料默认短生命周期

专项能力链优先把资料处理成：

- 临时材料
- `pending material`
- 会话级可消费材料

默认不自动正式入长期知识库；正式 `commit` 仍需用户确认。

### 原则 4：专项重处理与 Agent 主链分工明确

- 视频、网页、大文档等重处理：优先走专项能力链 / 异步控制平面
- 复杂分析、比较、组织与签字：优先走 Agent 主链

Agent 主链不应再长成第二套长后台任务系统。

### 原则 5：Executor Profile 与共享检索

- **`fast` / `complex` / `async`** 视为 **executor profile**（执行档位），不是抽象概念：
  - `fast`：小模型、少工具、无 planner
  - `complex`：大模型、多步骤、planner 开启（实现上为 Main→Middle→Answer）
  - `async`：聊天主链视角为 fast/complex 的**退出口**；专项重处理链可为正式 profile
- **`complex_candidate`** 由 ingress / `complexity_policy` 产出结构化信号；**不**强制初始 profile=complex
- **升级** = 切换 executor profile（如 fast→complex），不是「增加 Agent 数量」
- **KB 检索**在 fast/complex 之前作为共享步骤，产出 `retrieval_snapshot`；complex 升级后**不**自动再 retrieve，仅 `need_more_material` 时补 retrieve
- **`quality_gate`** 只判交付/升级，**不改 lane**；trace 必带 `profile_exit_reason`（async 退出时）
- **`approval_gate`** 独立于 `feedback_gate`；第一版仅覆盖 pending commit、长视频/重处理确认
- **二轮 refine 触发源** = `quality_gate.need_second_round`；`feedback_gate` 仅批准执行动作
- **`multi_source_compare` 无豁免**：与普通 complex 相同，round0 答 → `quality_gate` → `run_feedback_round_execution`；禁止 AnswerAgent 自触发二轮
- **approval_gate 与 commit 闭环**：用户确认保存且存在 pending 时，由 `approval_gate_flow.try_execute_commit_turn` 执行 commit 并早退，不再依赖 Main/Middle 旧 commit 链
- **material trace 全路径**：任意 turn 出口（fast / complex / async / approval_blocked / commit_executed）必须带 `material_layer_used` / `material_scope` / `material_source_count`

## 一、默认路由规则

### 1.1 `complex candidate` 的判定方式

`complex candidate` 按**任务形态**判定，不按资料类型判定。

也就是说：

- 视频问题不天然复杂
- `KB` 问题不天然复杂
- 关键在于用户要求的任务形态

### 1.2 默认视为 `complex candidate` 的题型

- 比较类：对比、异同、优缺点、适用场景比较
- 取舍 / 决策类：优先级、推荐、权衡、排序
- 多维分析类：从多个角度、多个维度、分情况展开
- 正反论证类：支持与反对、双边论证、辩证分析
- 跨材料综合类：结合多份知识、文档、证据综合判断
- 方案设计 / 路线图类：设计方案、整改路径、阶段计划

### 1.3 默认不视为 `complex candidate` 的题型

- 单一事实问答
- 单术语解释
- 单资料总结
- 低推理负担的简单列点题

### 1.4 `fast -> Agent` 的升级原则

对 `complex candidate`：

- 允许先尝试 `fast`
- 但只要 `fast` **不明显优秀**，就应升级到 Agent 主链

这里的“不明显优秀”在第一版中先按以下信号理解：

- 结构完成度不高
- 没有真正做判断 / 比较 / 取舍
- 证据支撑不足
- 回答过短、过浅、过于压缩
- limitations 明显
- 出现截断、异常输出或其他不稳定现象

## 二、材料流规则

### 2.1 三层材料状态

当前项目中的材料状态按以下三层理解：

1. `temporary material`
2. `pending material`
3. `committed knowledge`

### 2.2 默认材料流

默认流转是：

专项能力链
-> 临时材料 / `pending material`
-> `KB retrieve` / Agent 消费
-> 用户确认后才允许正式 `commit`

### 2.3 Agent 消费哪一层

Agent 默认应消费：

- 临时材料
- `pending material`
- 已提交知识库中的正式知识

Agent 不应以“等待长后台任务完成”作为自己的常规工作方式。

### 2.4 复杂 compare 的理想流向

对于重资料 compare，理想流向不是：

Agent 直接等待两个后台任务完成。

而是：

专项链先把资料处理成可消费材料
-> 材料进入临时池或 `pending`
-> Agent 基于这些材料做比较与组织

## 三、质量门控规则

### 3.1 门控总原则

质量门控不是单一总分，而是先看三类信号：

1. 任务完成度
2. 证据充分度
3. 答案质量

### 3.2 任务完成度

重点检查：

- 是否覆盖了用户显式要求的维度
- 是否满足格式与结构要求
- 是否遗漏关键子任务

例：

- 用户要求“三个角度”，答案只覆盖两个角度 -> 未达标
- 用户要求“正反论证”，答案只写一边 -> 未达标

### 3.3 证据充分度

重点检查：

- `kb_hits`
- `kb_top_score`
- `kb_evidence_tier`
- `rag_miss`
- `kb_adequate`
- 是否主要依赖低质量 fallback

### 3.4 答案质量

重点检查：

- 是否有结构
- 是否做了判断而不只是摘抄
- 是否处理了不确定性
- 是否存在截断、异常输出、明显空泛

## 四、`KB` 证据充分度规则

### 4.1 简单题：实用优先

对简单题：

- 证据大致够即可答
- 可以带 limitations
- 不要求过高证据门槛

### 4.2 复杂题：保守优先

对复杂题：

- 证据稍弱就不应直接交付
- 更倾向升级 Agent
- 或补材料
- 或明确说明证据不足

### 4.3 第一版建议门槛

以下数值仅作为第一版建议值，后续需结合评测再校准。

#### 简单题

- `kb_hits >= 1~2`
- `kb_evidence_tier >= medium`
- 无明显 `rag_miss`

#### 复杂题

- `kb_hits >= 3`
- `kb_evidence_tier = strong`
- 无 `rag_miss`
- 不应以低质量 fallback 作为答案主支撑

## 五、协作轮数升级规则

### 5.1 默认协作形态

默认协作形态为：

`Main -> Middle -> Answer`

先跑单轮。

### 5.2 第二轮的定位

第二轮 `refine` 不是奖励，而是补救与增强。

复杂题不对单轮结果过度乐观；只要质量不过线，就允许更积极地进入第二轮。

### 5.3 第二轮优先目标

第二轮应当：

- 两者都做
- 但**优先补材料**

也就是：

1. 先解决材料不足、证据偏弱、信息覆盖不全
2. 再优化结构、比较、判断与结论

### 5.4 第二轮触发条件

第一版建议把以下情况视为二轮触发条件：

- 输出未完成
- 用户要求的维度未覆盖全
- 材料不足
- 证据冲突
- 结论不稳
- 回答偏浅、偏压缩、综合不够

## 六、时延建议值

以下为第一版建议值，不是硬 SLA。

### 6.1 `fast`

- 首答目标：`2~5s`

### 6.2 Agent 单轮

- 同步完成目标：`10~20s`

### 6.3 Agent 第二轮

- 同步总时长建议上限：`20~30s`

### 6.4 超过 `30s`

若复杂题同步总时长预计超过 `30s`，应非常谨慎，优先考虑：

- 缩小第二轮范围
- 分阶段回答
- 或异步化

当前默认配置已把聊天主链同步预算上调到 **30 秒**，用于避免复杂题在刚进入 `complex` 后被过早截断成 `partial`。

## 七、并发原则

### 7.1 可以并发的部分

- 材料获取
- `KB retrieve`
- web evidence gather
- 候选证据准备
- 某些可拆分的局部子分析

### 7.2 不宜并发的部分

- 最终判断与统一收口
- 依赖第一轮结果的第二轮 `refine`

因此，复杂题的合理形态应是：

- 前半段尽量并发 gather
- 后半段串行收口

## 八、待后续实验校准的参数

以下参数已确定方向，但数值和边界仍需后续实验校准：

- `fast -> Agent` 的精确硬门槛 / 软门槛
- `KB` 证据充分度的精确数值阈值
- 哪些题型一定直接升级，而不必先试 `fast`
- 第二轮 `refine` 的更细 reason code
- 哪些复杂题允许同步逼近 `30s`

## 九、一句话冻结结论

当前项目的默认工作方式正式冻结为：

默认先 `fast`；`complex candidate` 只要 `fast` 不明显优秀就升级到 Agent；材料优先以临时池 / `pending` 形式供 Agent 消费；复杂题以质量优先，单轮不足时更积极地进入第二轮 `refine`，其中第二轮优先补材料，再做结构与结论优化。
