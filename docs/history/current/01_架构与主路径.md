# 01 架构与主路径

## 系统定位

这是一个有状态的多 Agent / RAG / Tool 编排系统，不是无状态聊天壳子。

当前现状仍以单轮主链为主要运行入口，但项目的**冻结目标架构**已经切换为：

- 外层平台图
- 内层三强自治核心

冻结口径见 [04_目标架构冻结与术语表](/D:/1/A1_publish/docs/current/04_目标架构冻结与术语表.md)。

当前代码现实下，更贴近的主流程是：

用户请求  
→ `POST /chat/agno`  
→ ingress 判断 `lane / mode / complex_candidate`  
→ approval_gate  
→ decision_arbitrator  
→ shared_material_prep  
→ `fast / complex / async`  
→ 如进入 `complex`，再走 `MainAgent -> MiddleAgent -> AnswerAgent`  
→ turn_exit_gate 统一收口  
→ `answer + extra + trace`

## 架构分层

### 接口层

- `backend/api/`

### 聊天主链编排

- `backend/application/chat/`（单轮编排入口）
- `backend/services/`（如 `agno_chat_service` 对外装配）

### Agent 层

- `backend/agents/main_agent/`
- `backend/agents/middle_agent/`
- `backend/agents/answer_agent/`

### 知识层

- `backend/knowledge/`

### 工具与检索

- `backend/tools/`
- `backend/rag/`、`backend/retrieval/`

### 存储层

- `backend/storage/`

## 当前收口点

当前主链里几个关键收口点：

- 公共出口状态：`backend/application/chat/turn_exit_gate.py`
- 材料层语义：`backend/application/chat/material_flow.py`
- 资料生命周期：`backend/services/capabilities/knowledge/pending_ingestion_service.py`

也就是说：
- Agent 不是唯一判断入口
- 但 complex 协作链内部仍然由三 Agent 负责复杂任务处理

## 目标终局说明

当前 `POST /chat/agno -> Main -> Middle -> Answer` 不能再作为整个系统的现行主入口事实。

更准确地说：
- `Main -> Middle -> Answer` 是 `complex` 主链事实
- 不是所有请求的默认入口事实

项目后续迁移必须回扣：

- 外层：`Semantic Router + Lane + Shared Capability Plane + Async Control Plane`
- 内层：`MainAgent + MiddleAgent + AnswerAgent`

也就是说：

- 当前运行事实是“外层先判、complex 内层再协作”
- 平台图 + 三强自治核心是冻结后的目标事实
- 后续 Phase 2 及之后的目录与运行时工作，都必须围绕这个目标推进
